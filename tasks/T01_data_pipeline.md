# T01: 解释 token、batch 和 loss 到底在优化什么

## 背景

团队里已经有人开始质疑这个项目: “看起来不就是把文本喂进去然后算个 loss 吗?” 如果你不能解释数据是怎么进模型、loss 为什么有意义，后面讨论模型能力时就会反复陷入空谈。

## 任务目标

- 可视化一条训练样本从原始文本到 `(x, y)` 的构造过程
- 解释 batch 切片如何影响模型看到的上下文
- 让读者理解 loss 对应的真实含义

## 业务约束

- 结果必须能给非算法工程师看懂
- 图和样例优先于公式
- 实验必须可以在极小数据集上复现

## 你需要改的代码

- `code/stage1_nanogpt_core/data.py`
- `code/stage1_nanogpt_core/train.py`
- `code/stage1_nanogpt_core/debug_batch.py`

## 交付标准

- 输出至少 1 个原始文本到 token 序列的示例
- 输出至少 1 组 `(x, y)` 对照样例
- 解释一个 batch 中每个位置的预测目标
- 说明 loss 下降为什么代表“更会预测下一个 token”

## 原理补充

对工程师来说，这一单真正要建立的是三件事:

1. token 不是“单词”，而是模型处理文本的离散单位
2. 一个训练样本本质上是“前文 -> 下一个 token”的大量并行预测
3. 平均 loss 不是抽象分数，而是“模型对正确下一个 token 的不确定程度”

这里不要急着讲 attention。先把数据流讲明白，后面读者才知道模型为何需要上下文表示能力。

## 实验步骤

1. 选择一段 20 到 40 个 token 的文本
2. 打印 token id 和可读文本片段
3. 展示 `x` 和 `y` 的错位关系
4. 在训练前后比较同一位置的 top-k 预测

## 结果解读

如果读者能看懂“模型在每个位置都在猜下一个 token”，他就会开始理解:

- 为什么大模型能从 next-token prediction 中学到语言结构
- 为什么更多上下文往往有帮助
- 为什么低 loss 不必然等于高层任务都做得好

## 线上风险

- 团队误把 loss 当最终业务指标
- 数据清洗问题被模型问题掩盖
- batch 切片方式不一致导致实验不可比
