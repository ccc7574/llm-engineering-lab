# C01: 让模型理解仓库级上下文，而不只是单文件补全

## 背景

团队做了一轮代码补全 demo 后，马上发现单文件补全的价值有限。真实问题通常需要知道:

- 这个函数在哪被调用
- 相关类型定义在哪
- 测试怎么写
- 这个仓库的风格和约定是什么

这说明 coding 能力的关键瓶颈往往不是模型本身，而是上下文构造。

## 任务目标

- 定义 repo-level context 的最小打包策略
- 比较“只看当前文件”和“带仓库上下文”的差异
- 建立后续 bugfix 和 agentic coding 的上下文底座

## 业务与资源约束

- 上下文长度有限
- 不能把整个仓库无脑塞进 prompt
- 必须考虑检索质量和上下文预算

## 代码改造点

- `code/common/`
- `eval/`
- 可新增 `code/stage_coding/context_builder.py`

## 交付标准

- 至少定义一种上下文打包格式
- 至少支持同文件、邻近文件、测试文件三类上下文片段
- 至少做一个前后对照实验
- 至少记录上下文召回率或上下文预算

## 原理补充

repo-level context 的核心不只是“给更多 token”，而是:

1. 给模型正确的信息
2. 在有限窗口内给最相关的信息
3. 让上下文顺序、结构和标记方式可控

## 实验步骤

1. 选一个需要跨文件理解的小任务
2. 分别构造无上下文、弱上下文、强上下文三组输入
3. 对比代码补全或 bugfix 结果
4. 记录有效信息密度与上下文预算
5. 观察 `retrieved` 是否真的选中了关键文件

## 结果解读

读者应该能形成判断:

- 上下文构造是 coding model 工程能力的核心部分
- 很多“模型不行”的现象，其实是上下文给错了

## 线上风险

- 检索召回太多无关代码
- 上下文排序混乱，反而误导模型
- 测试和实现版本不一致，导致模型学到错误约束
