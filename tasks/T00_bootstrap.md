# T00: 接手遗留最小模型并跑通训练闭环

## 背景

你加入团队第一天，前任工程师留下一套最小 GPT 代码。它能训练、能采样，但没有整理好的运行说明，也没有人敢保证它现在还能正常跑。

本周五要做内部分享，你至少需要让系统在一台开发机上完成一次最小训练和文本生成，证明这个项目还有继续演化的价值。

## 任务目标

- 跑通最小训练闭环
- 产出一段可展示的生成结果
- 给后续任务建立统一运行入口

## 业务约束

- 默认只有单机环境
- 优先 CPU 或单卡可跑通
- 30 分钟内能完成最小实验
- 不追求效果，只追求链路完整

## 你需要改的代码

- `code/stage1_nanogpt_core/train.py`
- `code/stage1_nanogpt_core/sample.py`
- `code/stage1_nanogpt_core/config.py`
- `README.md`

## 交付标准

- 能启动一次最小训练
- 能加载 checkpoint 做采样
- README 中能清楚说明依赖、命令和输出位置
- 给出至少 2 组示例生成结果

## 原理补充

这一单不是为了讲深奥原理，而是让读者先明白语言模型最小闭环是什么:

1. 文本被切成 token
2. token 序列被送进模型
3. 模型预测下一个 token 的分布
4. 用交叉熵损失训练
5. 再把采样出的 token 拼回文本

如果这条链路跑不通，后面所有关于 attention、SFT、推理增强的讨论都没有抓手。

## 实验步骤

1. 准备一个极小文本数据集
2. 跑几百到几千 step 的最小训练
3. 固定随机种子采样
4. 保存一份初始输出和一份训练后输出

## 结果解读

这一阶段最重要的不是“像不像人话”，而是:

- loss 是否下降
- 采样是否明显区别于随机噪声
- 训练和推理脚本是否有最小可复现性

## 线上风险

- 训练脚本入口过多，后续同事不知道该用哪个
- 数据和 checkpoint 路径不统一，复现容易失败
- 只展示效果不展示命令，团队很快失去信任

## 这一单结束后读者应该获得什么

- 对最小语言模型闭环有操作层面的理解
- 知道后续所有能力增强都建立在这条链路上
- 愿意继续进入下一单，而不是被一开始的环境问题劝退
