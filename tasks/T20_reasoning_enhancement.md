# T20: 解决多步任务不稳定，给助手补上推理增强链路

## 背景

模型已经可以回答常规问题，但一遇到多条件判断、简单数学、规则推导或多步决策，正确率就开始波动。产品团队第一反应通常是“换更强模型”，但在预算有限时，你更需要先验证推理增强路线值不值得做。

## 任务目标

- 给模型增加多候选采样能力
- 引入 self-consistency 投票
- 设计最小 verifier / reranker 流程
- 让读者看到“推理模型”可以由一组工程机制逐步拼出来

## 业务约束

- 先做小规模验证，不追求 fancy 算法
- 每一步增强都要能量化收益和成本
- 必须明确哪些收益来自训练，哪些来自推理时策略

## 你需要改的代码

- `code/stage3_reasoning/sample_reasoning.py`
- `code/stage3_reasoning/self_consistency.py`
- `code/stage4_verifier/train_verifier.py`
- `code/stage4_verifier/rerank.py`
- `eval/reasoning_eval.py`

## 交付标准

- 同一题目能够生成多个候选解
- 能基于投票或评分选出最终答案
- 给出单次采样、多次采样、verifier 重排三者的对比
- 说明各方案的延迟、成本和效果差异

## 原理补充

这一单的关键是拆掉“推理模型很神秘”这个误解。

在工程实践里，很多所谓 reasoning enhancement 都可以拆成几类手段:

1. 让模型输出更完整的中间过程
2. 对同一问题采样多条候选路径
3. 用投票、一致性或 verifier 提高最终答案稳定性
4. 再进一步，才是用偏好优化或强化学习改变模型策略

也就是说，推理能力的提升不一定来自全新架构，常常来自:

- 数据格式变化
- 推理时搜索策略变化
- 结果筛选机制变化

## 实验步骤

1. 准备一个小型多步任务集
2. 对每题生成 1 条、3 条、5 条候选解
3. 比较 greedy、temperature sampling 和 self-consistency
4. 训练一个最小 verifier，对候选进行重排
5. 记录正确率、平均延迟和 token 开销

## 结果解读

这一单结束后，读者应该形成一种非常务实的判断:

- 推理增强不是非黑即白
- 很多场景可以先用采样加重排获得明显收益
- 只有在这些手段不足时，再去考虑更贵的训练路线

## 线上风险

- 中间过程更长，但答案并没有更准
- 多采样成本过高，线上不可接受
- verifier 过拟合，离开小任务集后失效
- 团队把“能生成步骤”误判为“真的会推理”
