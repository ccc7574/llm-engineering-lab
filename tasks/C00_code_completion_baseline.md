# C00: 建一个代码补全最小基线，理解代码模型和自然语言模型的差异

## 背景

团队准备做代码生成或代码补全能力，但工程师内部有一个常见误区: “代码不也是文本吗，直接拿通用模型改一改就行了?” 如果这个问题不先讲清楚，后面的 repo context、bugfix、execution feedback 都会被低估。

## 任务目标

- 建立代码补全任务的最小闭环
- 展示代码补全和自然语言续写在数据分布上的差异
- 给后续 coding 能力线建立统一基线

## 业务与资源约束

- 优先单机可跑
- 先做最小 Python 代码样本，不追求真实大规模代码模型
- 先关注任务定义和评测，不追求模型性能上限

## 代码改造点

- `code/stage1_nanogpt_core/`
- `datasets/`
- `eval/`
- 后续可新增 `code/stage_coding/`

## 交付标准

- 至少定义一种代码补全输入输出格式
- 至少准备一个极小代码样本集
- 至少提供一个 baseline 推理脚本
- 至少提供一个最小评测指标

## 原理补充

这里最重要的不是“代码模型更高级”，而是让读者理解:

1. 代码 token 分布比自然语言更结构化
2. 代码任务的正确性常常不能只靠表面流畅度判断
3. coding eval 很多时候需要 execution-based feedback

## 实验步骤

1. 准备一组简单函数补全样本
2. 让 baseline 模型做 suffix completion
3. 比较自然语言样本和代码样本的续写表现
4. 记录 exact match 和可执行率

## 结果解读

这单结束后，读者应该知道:

- 为什么代码补全是独立能力，不是普通续写的直接延伸
- 为什么 coding 能力线必须单独做数据与评测设计

## 线上风险

- 用自然语言指标误评代码生成
- 只看样例，不看执行结果
- 忽略语法正确但逻辑错误的情况
