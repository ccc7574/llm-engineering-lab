# T10: 把 Base Model 改造成能按要求回答的助手

## 背景

当前模型已经能续写，但产品团队真正需要的是一个“听得懂要求、按格式回答”的内部助手。只做预训练不会自然得到这种行为，你需要给团队一个最小可运行的 SFT 路线。

## 任务目标

- 引入指令模板
- 设计最小 SFT 数据格式
- 让模型从“继续写文本”变成“按角色回答问题”

## 业务约束

- 数据规模可以很小，但格式必须规范
- 训练脚本不能比预训练脚本复杂一个数量级
- 要能清楚解释 base model 和 instruction model 的区别

## 你需要改的代码

- `code/stage2_sft/train_sft.py`
- `code/stage2_sft/dataset.py`
- `code/stage2_sft/prompt_template.py`
- `eval/sft_eval.py`

## 交付标准

- 定义统一的 instruction 数据模板
- 支持只对 assistant 回答部分计算 loss
- 给出 base model 与 SFT model 的对照案例
- 至少包含 1 组结构化输出任务

## 原理补充

这里要让读者理解三件事:

1. base model 学到的是“如何继续文本”
2. SFT 让模型学会“在特定交互协议下回答”
3. 所谓助手化，本质上是让模型适应新的输入输出分布和行为期望

对工程师来说，最关键的代码点往往不是网络结构，而是:

- prompt 模板
- 数据清洗
- label mask
- 评测样本

## 实验步骤

1. 组织一个极小 instruction 数据集
2. 训练 base model 和 SFT model
3. 对比两者在问答、摘要、结构化输出上的表现
4. 观察错误类型是否从“不会回答”变成“回答不稳定”

## 结果解读

这一单结束后，读者应该知道:

- 为什么大部分应用模型不是直接拿 base model 上线
- 为什么很多“模型变聪明了”的现象，其实是行为分布被重新塑形
- 为什么 SFT 常常是推理增强前的必要底座

## 线上风险

- 模板不稳定，线上 prompt 和训练 prompt 不一致
- label mask 错误，模型学会复述用户输入
- 只看少量样例就误判 SFT 成功
