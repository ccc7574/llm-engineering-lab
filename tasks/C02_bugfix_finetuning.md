# C02: 用最小 bugfix 数据做 coding 微调，理解代码行为塑形

## 背景

只做代码补全不够支撑真实工作。真实团队更关心的问题是:

- 给一个 bug report，模型能不能修
- 给一段失败测试，模型能不能定位问题
- 给一个错误实现，模型能不能往正确方向改

这类任务更接近代码行为塑形，而不仅是续写。

## 任务目标

- 构造最小 bugfix 训练样本
- 让模型从“续写代码”走向“按约束修代码”
- 建立 coding SFT / post-training 的最小实验路径

## 业务与资源约束

- 样本规模可以很小，但输入输出协议要规范
- 必须区分 bug 描述、上下文代码和目标补丁
- 先不追求仓库级全量修复

## 代码改造点

- `datasets/`
- `eval/`
- 可新增 `code/stage_coding/train_bugfix_sft.py`

## 交付标准

- 至少定义一个 bugfix 数据格式
- 至少完成一轮最小训练或 imitation 实验
- 至少给出训练前后样例对照

## 原理补充

bugfix 微调和普通代码续写的关键区别在于:

1. 输入通常带问题约束
2. 输出通常要求局部修改，而不是自由生成
3. 评测更依赖测试、diff 和语义正确性

## 实验步骤

1. 选几个简单函数错误样本
2. 组织 `issue + code + expected fix` 数据
3. 训练或模拟一轮 coding SFT
4. 比较前后输出差异

## 结果解读

读者应该知道:

- coding 后训练更像行为约束学习
- 代码模型的数据组织方式会直接影响修复质量

## 线上风险

- 模型用大改动掩盖小 bug
- 生成的 patch 表面合理但破坏其他行为
- 样本太单一，模型只学会模板化修复

## 复盘与延伸

- `C03` execution feedback
- `C10` coding judge / verifier
