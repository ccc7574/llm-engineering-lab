# A01: 用多步工具链完成任务，理解 planner-executor 的最小形态

## 背景

单工具调用只能证明 agent 会“拿外部信息”，但还不足以体现真实工作里的复杂性。很多任务需要多步串联:

- 先查信息
- 再做计算
- 再生成最终动作或回复

这就是多步 agent 任务的最小形态。

## 任务目标

- 建立多步工具链任务
- 记录 agent 的中间步骤和状态
- 让读者理解为什么 agent eval 不能只看最终结果

## 业务与资源约束

- 先做确定性工具链
- 先不引入复杂规划算法
- 每一步都要可解释、可回放

## 代码改造点

- `code/stage_agentic/task_runner.py`
- `code/stage_agentic/tools.py`
- `eval/agentic_eval.py`

## 交付标准

- 至少有一类两步以上工具链任务
- 至少记录每一步 trace
- 至少统计平均 steps 与 average tool calls

## 原理补充

多步工具链的工程意义在于:

1. 把复杂任务拆成可执行动作
2. 让状态在步骤之间流动
3. 让错误可以被定位到具体一步

## 实验步骤

1. 设计一个必须多步完成的任务
2. 跑 direct 策略和 tool_use 策略
3. 比较最终 success、steps、tool_calls
4. 检查 trace 是否可解释

## 结果解读

读者应该知道:

- agent 的价值往往来自多步执行，而不是单次回答
- 过程指标和结果指标必须一起看

## 线上风险

- 工具链过长，延迟和失败率上升
- 中间状态不规范，导致后续步骤出错
- 没有 trace，导致定位问题困难
