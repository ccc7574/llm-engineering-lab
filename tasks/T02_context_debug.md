# T02: 向团队证明模型真的在利用上下文

## 背景

产品经理看完 demo 后提出一个尖锐问题: “它是不是只是背常见续写?” 如果你不能证明模型会根据上下文动态调整输出，团队会倾向于把所有现象都归结为记忆或模板匹配。

## 任务目标

- 用最小实验展示同一个前缀变化会如何改变预测
- 做 attention 可视化，帮助团队理解“按内容取信息”的机制
- 建立“上下文窗口是资源”这一认知

## 业务约束

- 不要求严谨论文级解释
- 必须对产品和工程同学都直观
- 图示和对照实验优先

## 你需要改的代码

- `code/stage1_nanogpt_core/model.py`
- `code/stage1_nanogpt_core/sample.py`
- `code/stage1_nanogpt_core/visualize_attention.py`

## 交付标准

- 给出至少 2 组上下文对照样例
- 输出至少 1 张 attention 热力图
- 解释上下文长度变化对生成的影响
- 明确区分“记住统计模式”和“利用当前上下文”这两件事

## 原理补充

这里开始引入 attention，但仍然用工程语言来讲:

- embedding 让 token 从离散 id 变成可计算向量
- attention 让当前 token 位置去“查找”哪些历史位置更相关
- causal mask 确保模型只能看见过去，不能偷看未来

对工程师最有帮助的理解是:

> attention 可以看成一种按内容寻址的上下文读取机制，模型不是把过去所有 token 等权相加，而是在每一步动态决定“现在该看哪里”。

## 实验步骤

1. 设计同一问句、不同上下文的对照输入
2. 比较模型在关键位置的 logits 或 top-k
3. 输出某一层、某一头的注意力图
4. 缩短上下文窗口，再重复同一实验

## 结果解读

这一单结束后，读者应该形成两个关键判断:

- 模型不是简单背答案，而是在做上下文条件化预测
- 上下文窗口和 attention 质量会直接影响复杂任务表现

## 线上风险

- 团队误把热力图当“因果解释”
- 只看一两个 head 就下结论
- 不区分训练数据记忆和推理时上下文利用
