# H03: 给模型链路补上 latency、token 和 cost profiling

## 背景

很多模型实验在离线评测里看起来提升明显，但一接近真实使用就会遇到新问题:

- 延迟太高
- token 消耗太大
- verifier 或多采样把成本拉爆
- 整体吞吐不满足线上要求

如果不把这些工程指标引入日常评测，团队很容易做出无法落地的优化。

## 任务目标

- 为训练后与推理链路增加 latency、token、cost profiling
- 比较不同增强策略的真实工程代价
- 建立“效果提升必须配合成本视角”的工程习惯

## 业务与资源约束

- 优先统计最关键指标
- 结果要便于和能力指标一起阅读
- 先支持 tiny 项目，再扩展到更复杂链路

## 代码改造点

- `eval/`
- `code/stage3_reasoning/`
- `code/stage4_verifier/`
- 可新增 profiling 工具

## 交付标准

- 至少记录平均生成 token 数
- 至少记录平均候选数或 verifier 调用数
- 至少能给出不同策略的相对成本对比
- 至少能把成本指标和质量指标放进同一张 summary board

## 原理补充

工程上真正有价值的优化，往往不是“效果最高”，而是:

1. 在可接受成本下效果更高
2. 在固定成本下效果更稳
3. 在固定效果下更便宜、更快

## 实验步骤

1. 比较 greedy、self-consistency、verifier rerank
2. 记录能力指标
3. 同时记录 token、latency、调用次数
4. 输出一个统一对照表

## 结果解读

读者应该知道:

- 模型优化永远不是只看准确率
- 工程指标决定一条路线能不能被团队接受

## 线上风险

- 只追求准确率，忽略成本
- 指标统计口径不一致，导致误判
- 没有 baseline，无法判断额外开销是否值得
